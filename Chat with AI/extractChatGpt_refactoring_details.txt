The reason it appears in the current code is that the foreman does not ask the supplier:

> “Give me the slab immediately after `currentSlab`.”

It asks something closer to:

> “Give me all currently selected slabs in this ready deck, except the exact current DOM element.”

That creates the issue.

Example with one deck containing slabs `A, B`:

1. Extract `A`; set `current = A`.
2. Re-query deck: `A` is skipped because `el === current`; `B` is selected.
3. Extract `B`; set `current = B`.
4. Re-query deck: now `B` is skipped, but `A` is no longer `current`, so without `seenIds`, `A` can be selected again.

So `seenIds` compensates for the fact that the supplier query is **set-like**, not **successor-like**.

Contiguity would help if the code used it to establish order and say:

> “Given the ordered slabs in this deck, pick the first slab after `currentSlab`.”

Then dedup would be much less important, maybe only a defensive check against remounts or duplicate DOM wrappers.

There is also a separate deck-level dedup: ChatGPT can expose multiple DOM containers with the same `data-turn-id`. That is not about slab order; it is about the supplier offering duplicate wrappers for the same logical deck.

So I’d frame it this way:

- Conceptually, the architecture does **not** need dedup as a first-class idea.
- The current implementation needs `seenIds` because it repeatedly asks for a filtered set of slabs, not a true successor.
- A cleaner future refactor would move toward a supplier operation like `nextSlabAfter(currentSlab, readyDeck)`.
- Then dedup becomes a defensive sanity guard, not part of the core flow.

- Supplier exposes unordered/weakly ordered candidates.

  nextSlabAfter(currentSlab, readyDeck)

  - re-querying;
  - geometry;
  - ordering;
  - current lookup;
  - duplicate DOM wrappers;
  - selector filtering.
- Core only sees:
  ```text
  currentSlab → nextSlab or no slab in this deck
  ```
- Cleaner core, but stronger supplier abstraction.

Given the architecture we are building, I prefer **Option B** as the target model:

> The foreman asks the supplier for the next slab. The supplier decides how to derive successor order from the changing DOM supply surface.

Then dedup becomes supplier-side implementation detail or diagnostic defense, not a core concept.

- The function `findNextPromptIn(...)` is already supplier-ish.

> The exact boundary between foreman and supplier is a design choice. - It may let us remove ancestor/descendant filtering.

  - same-turn-id duplicate handling, or
  - a stronger canonical-deck selection rule.

function queryTopLevelDeckCandidates() {
  return [...document.querySelectorAll(

- inner section also marked with the same turn id;

- We should not query “containers.”
- We should query **deck markers** or **deck-marked elements**.

> topmost deck-marked element for each marked subtree

> top-level deck container.

queryTopDeckMarkers()
