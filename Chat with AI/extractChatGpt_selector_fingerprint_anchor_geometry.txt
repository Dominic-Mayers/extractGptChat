`4 → 3 → 2 → 1 → 0`

- `0` while an unextracted message actually overlaps the container, or

ASSISTANT:
It can be `2` when one `readyContainer` contains two anchored messages.

The function does this:

```js
for (const el of document.querySelectorAll('[data-message-author-role]')) {
    ...
    if (probe > r.top && probe <= r.bottom) candidates.push(el);
}
```

So it scans **all mounted message elements**, then asks:

“Is this message’s probe point geometrically inside the current ready container?”

If two message elements have probe points inside the same container, both are pushed.

Conceptually:

```text
readyContainer
  ├─ assistant message A
  └─ assistant message B
```

On the first scan:

```text
A unseen + inside container → candidate
B unseen + inside container → candidate
```

So `candidates.length === 2`.

Then the loop consumes only `candidates[0]`. On the next scan, `A` is now in `seenIds`, so it is skipped, while `B` is still unseen and still inside the same container:

```text
A seen → skip
B unseen + inside container → candidate
```

So the sequence is:

```text
2 → 1 → 0
```

This is why `seenIds` is necessary even inside one container.

- Current code mostly does **not** do this during slab selection.

- Once the deck is ready, the code assumes the deck’s contents can be drained, even if some slabs are not currently visible.

That second point is important. If we restricted slabs to the visible/lit viewport, a large ready deck could be only partially drained. The code’s model is:

> viewport/light activates the deck; deck geometry defines membership.

Not:

> viewport geometry defines which slabs are extractable.

So the cleaner formulation is:

- Query broad DOM candidate pool.
- Immediately filter by **deck geometry**.
- Do not filter by viewport geometry after deck readiness, unless we decide readiness is only local to the lit slice.

The bug we are seeing is not mainly “missing immediate geometry restriction.” It is that the anchorless branch uses the wrong kind of DOM object as a slab candidate: `[data-turn]` / deck-ish DOM instead of known slab DOM.

## My request for Codex:
If we consider only the core, perhaps we only need to interact with the DOM to ask the lit slabs in the deck, filter by selector, and wait by fingerprint(s) ?  What else ?


========================================

ASSISTANT:
Yes — for the **core traversal/extraction model**, that is very close.

I’d phrase the core DOM interactions as four operations:

1. **Pick / advance deck**
- Bootstrap: pick edge-visible deck.
- Main loop: pick adjacent deck by geometry.
- DOM role: deck navigation.

2. **Wait for deck readiness**
- Bring deck into light.
- Wait for `data-is-intersecting` / deck-level fingerprint.
- DOM role: external-worker activation for the deck.

3. **Ask for lit slabs in deck**
- Query candidate slab/anchor DOM.
- Geometrically restrict to current ready deck.
- Apply selector: anchored message, anchorless extractable Markdown, known slab classes.
- DOM role: slab discovery.

4. **Wait for slab readiness and extract**
- Apply slab finish fingerprint if needed.
- Serialize selected slab once.
- DOM role: content readiness + extraction.

Everything else is supporting/diagnostic, not core:

- nav dots / TOC count: progress validation and edge jump convenience;
- scroll-height stability: bootstrap safety;
- probe misses / model checks: diagnostics;
- mutation watchers: fingerprint discovery;
- delayed rechecks: diagnostic only;
- image downloading: export post-processing;
- gap/inset/slab-stack checks: model verification, not traversal;
- performance timers: observability.

So yes, the core can be reduced to:

> choose deck → wait deck ready → enumerate slab candidates in deck → select extractable slabs → wait slab ready → extract.

One extra core-ish operation remains:

5. **Identity / dedup**
- Maintain consumed slab/message ids.
- Needed because the same ready deck can be queried multiple times.
- Without this, multi-slab decks or repeated scans can duplicate extraction.

So the minimum core is really:

- deck navigation;
- deck readiness;
- slab enumeration/selection;
- slab readiness;
- extraction;
- identity/dedup.

## My request for Codex:
Perhaps we should separate DOM (root), decks, slabs, and viewport. The model must specify the properties of each. In this way, deck readiness is a property of decks, only indirectly of the DOM.  


========================================

ASSISTANT:
Yes — that separation is the missing formal layer. I’d define four objects:

1. **DOM Root**
- The observable universe.
- Provides queries for mounted decks/slabs/anchors.
- Has no readiness of its own.
- Properties: mounted elements, document order, structural ancestry, mutation events.

2. **Viewport / Light**
- The activation window.
- External workers react to things entering it.
- Properties: top/bottom geometry, direction of travel, scroll position.
- It does not contain transcript truth; it only triggers/permits realization.

3. **Deck**
- A top-level traversal region.
- Corresponds to a mounted `[data-turn-id-container]` / `section[data-turn]` region.
- Properties:
  - identity: `turnId`;
  - geometry: rect;
  - adjacency to other decks;
  - readiness: deck fingerprint, e.g. `data-is-intersecting`;
  - slab membership: slabs geometrically or structurally belonging to it.

4. **Slab**
- An extractable content unit inside a deck.
- Properties:
  - identity: `messageId` if anchored, otherwise `turnId`/derived identity;
  - role: user/assistant/anchorless;
  - selector validity: “is this an actual slab, not control DOM?”;
  - readiness: slab finish fingerprint;
  - content: serializable Markdown;
  - geometry: rect/scope.

Then the core becomes almost declarative:

```text
DOM Root provides mounted decks.
Viewport lights decks.
Deck readiness belongs to decks.
Decks provide slab candidates.
Slab selector decides whether a candidate is a real slab.
Slab readiness belongs to slabs.
Extraction consumes ready slabs.
```

This also prevents category mistakes:

- `data-is-intersecting` is not a DOM-root property; it is a **deck readiness fingerprint**.
- Markdown presence is not readiness alone; it can be a **slab selector**.
- `data-message-author-role` is not necessarily the slab’s whole region; it is an **anchor/selector signal**.
- The viewport is not the deck; it only lights or activates decks.

I think this should go into the README as the conceptual model, because it would have prevented several of our confusions.
