## My request for Codex:
This makes me think about the question how the driven adapters of an application  relate to the phases of the universal installer. 


========================================

ASSISTANT:
Yes — there is a strong parallel.

In an application, **driven adapters** are the concrete things the core calls outward to get work done: filesystem, network, DOM, package manager, browser, etc.

In the universal installer, the phases are probably not just “steps.” They are moments where the core needs different kinds of driven adapters.

A rough mapping:

- **Discover**  
  Adapter answers: “What exists?”  
  Example: enumerate decks, files, packages, installed tools.

- **Inspect / Measure**  
  Adapter answers: “What is its current state?”  
  Example: geometry, readiness flag, version, checksum, installed marker.

- **Prepare / Resolve**  
  Adapter answers: “Can you make it ready?”  
  Example: scroll into view, download dependency, create directory, hydrate component.

- **Verify**  
  Adapter answers: “Is it now usable?”  
  Example: readiness fingerprint, file exists, command returns expected version.

- **Extract / Apply**  
  Adapter performs the real action.  
  Example: serialize slab, install package, write config, run migration.

- **Diagnose**  
  Adapter provides evidence when the model fails.  
  Example: DOM dump, logs, diff, command output, geometry traces.

The key thought: phases are not merely chronological. They define **what kind of question the core asks the outside world**.

For this extractor:

```text
discover deck → measure geometry → prepare by scrolling → verify readiness
→ discover slabs → verify slab finished → extract markdown → diagnose gaps
```

So yes: the driven adapters of an application relate to installer phases by implementing the phase-specific questions/actions the core is allowed to ask. The phase model can help decide adapter boundaries.

ASSISTANT:
Yes. Moving intelligence into adapters is not automatically a loss. It is only a problem if the adapter becomes an unstructured one-off blob.

A better rule is:

```text
Adapters may be intelligent,
but their intelligence should be capability-shaped.
```

So an adapter can know a lot about ChatGPT’s DOM, just as a filesystem adapter can know a lot about paths, permissions, and atomic writes. That knowledge belongs near the boundary.

The danger signs are different:

- **Too big**: one adapter function does many unrelated supplier requests.
- **Too application-specific**: cannot be named as a reusable capability.
- **Too orchestration-heavy**: decides workflow order instead of answering a bounded request.
- **Too hidden**: exposes “do everything” instead of smaller inspect/measure/prepare/check operations.

For the extractor, good adapter capabilities might be:

```text
list deck sequence containers
measure deck geometry
find next deck by geometry
wait for deck readiness
list slabs in deck
check slab finished
extract slab markdown
diagnose deck/slab model
```

Some intelligence can live in each. For example, `findNextDeck` can be smart about geometry. That is fine if its capability is precisely:

```text
Given current deck and direction, return next deck.
```

But if it also waits for readiness, extracts slabs, and logs model violations, then it is too large.

So yes: adapters are runtime libraries. Good libraries can be smart; they just need clean exports.
