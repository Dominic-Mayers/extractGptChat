## My request for Codex:
Each cycle of the loop starts with what ?


========================================

ASSISTANT:
Each normal cycle starts with a **ready deck**.

More precisely, at the top of the main loop the code already has:

- `readyContainer`: the current deck;
- `current`: the last slab consumed;
- `seenIds`: slab/message identities already consumed;
- `containerInsertAt`: where additional slabs from this deck should be inserted.

Then the cycle does:

1. **Check/model the current ready deck**
   ```js
   checkReadyContainerModel(readyContainer, checkedModelContainers)
   ```

2. **Ask for the next unconsumed slab inside that ready deck**
   ```js
   const slabs = findNextPromptIn(readyContainer, current, seenIds)
   ```

So the cycle begins not by choosing a deck, but by assuming:

> “I have a ready deck; can I drain another slab from it?”

If yes: extract that slab and keep the same deck.

If no: the deck is drained, so the loop then advances to the adjacent deck and waits for it to become ready. That newly ready deck becomes the starting point of the next cycle.

## My request for Codex:
What is Check/model ?


========================================

ASSISTANT:
`checkReadyContainerModel(...)` is diagnostic, not core traversal.

It does not decide what to extract. It observes whether the current ready deck still matches our mental model.

It checks things like:

- Are anchored messages geometrically contained in the ready deck?
- Are there message anchors that overlap the deck but failed the probe-membership test?
- Are adjacent message anchors separated by suspicious gaps?
- Is the first message anchor far below the deck top?
- Is the last message anchor far above the deck bottom?
- What direct slab-like children exist in the deck’s content stack?
- Are there unknown slab-like children?
- Are slab-like children contiguous, or are there slab gaps?

So “check/model” means:

> run model-validation diagnostics on the ready deck before extracting from it.

It is not part of the minimal algorithm. The minimal loop could start directly with:

```js
const slabs = findNextPromptIn(readyContainer, current, seenIds)
```

The model check is there to catch violations of our assumptions while the extractor still continues when possible.

We should define the algorithm as two phases:

2. **Main loop**

```text
  choose edge-visible deck

  extract first slab

  readyDeck is selected and deck-ready
  current is the last consumed slab
  seenIds records consumed slab identities

Main loop:
  ask readyDeck for next unconsumed selected slab
  if found:

    extract
    update current and seenIds
  else:

    wait deck readiness
    set readyDeck = adjacent deck
```

That also makes `checkReadyContainerModel` easy to place:

```text
Optional diagnostic at main-loop start:
  inspect readyDeck model
```

not part of the invariant and not part of readiness itself.

## My request for Codex:
Yes this perhaps a subtle point in the bootstrap. How is it managed concretely (remaining as conceptual as possible in the explanation) ?


========================================

ASSISTANT:
Concretely, bootstrap creates the first cursor by doing one full normal extraction before the main loop starts.

Conceptually:

1. **Pick first deck**
   - At the starting viewport edge, choose the edge-visible deck.

2. **Make it ready**
   - Wait until that deck’s readiness fingerprint holds.

3. **Ask that deck for slabs with no cursor yet**
   - Bootstrap calls the same slab-discovery logic, but with:
     ```text
     currentSlab = null
     ```
   - Meaning: “there is no previous slab to skip; give me the first selected slab in this ready deck.”

4. **Extract first slab**
   - Wait for that slab’s readiness.
   - Extract it.
   - Record its identity in `seenIds`.

5. **Create the cursor**
   - Set:
     ```text
     currentSlab = first extracted slab
     readyDeck = first deck
     ```

Only after this does the main loop begin.

So the subtle point is:

> Bootstrap is allowed to query a ready deck without a current slab, but only once, to manufacture the first current slab.

After that, every main-loop cycle has a `currentSlab`.

When the loop later moves to a new deck, `currentSlab` still refers to the last slab in the previous deck. That sounds odd, but it is harmless because:

- the new deck is selected independently by deck geometry;
- slab discovery is constrained to the new ready deck;
- `currentSlab` only excludes exact same DOM element;
- `seenIds` handles real deduplication.

So `currentSlab` is partly a cursor, partly a “do not immediately repeat this exact element” guard. The durable transcript cursor is really the pair:

```text
readyDeck + seenIds
```

## My request for Codex:
Does the main loop has to be over deck. Could it be that once we have the first slab, we run over slabs and move viewport and move to the next deck as needed.  


========================================

ASSISTANT:
Yes, conceptually the main loop can be “over slabs.”

That may actually be the cleaner abstraction:

```text
Bootstrap:
  produce first slab and its deck.

Main loop:
  repeatedly produce the next slab.
```

Then “move to next deck if needed” becomes an implementation detail of `nextSlab()`.

The loop would be:

```text
currentSlab = bootstrapFirstSlab()

while currentSlab exists:
  extract currentSlab
  currentSlab = nextSlab(currentSlab)
```

And `nextSlab(currentSlab)` would internally do:

```text
1. Look for another unconsumed slab in current deck.
2. If found, return it.
3. If not found:
   a. choose adjacent deck;
   b. wait for deck readiness;
   c. look for first unconsumed slab in that deck;
   d. return it.
```

This fits the user-level goal better: export a sequence of slabs/messages.

The deck is then not the loop’s subject; it is the traversal medium. That is probably more natural architecturally:

- **Slab loop**: transcript sequence.
- **Deck machinery**: how the next slab is made available.
- **Viewport/light**: how decks become ready.
- **DOM root**: where candidates are queried.

Current code is deck-shaped because it physically drains a `readyContainer` and only then advances. But abstractly, it is a slab iterator with internal deck state.

A good architecture statement would be:

> The extractor’s public/core abstraction is an iterator over slabs. Internally, the iterator maintains a current ready deck; when no further slab is available in that deck, it advances the deck and resumes slab selection.

That may be the best way to resolve the “does the loop start with deck or slab?” ambiguity:

- bootstrap establishes first slab;
- main loop is over slabs;
- next-slab resolution may reuse current deck or move to adjacent deck.

So viewport movement is not an independent “scroll loop.” It is triggered by failure to find the next slab in the current deck.

A clean statement:

```text
The extractor iterates slabs.
Viewport movement is a side effect of resolving the next slab when the next slab is not available in the current ready deck.
```

And:

```text
Slab-level need: produce the next transcript unit.
Deck-level mechanism: locate and ready the adjacent deck.
Viewport-level mechanism: move light so external workers prepare that deck.
```

So yes: even though the code calls `bringIntoView` on decks, the computation belongs inside `nextSlab()` — because its purpose is to satisfy the next-slab request.
